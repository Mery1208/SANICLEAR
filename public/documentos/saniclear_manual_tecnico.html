Sección 01

01 Introducción

1.1 Descripción del Proyecto

SANICLEAR es una aplicación web de tipo Single Page Application (SPA) diseñada para la digitalización integral de los procesos de limpieza y desinfección en entornos hospitalarios. Su objetivo principal es sustituir los partes de trabajo físicos por una plataforma digital centralizada, accesible desde cualquier dispositivo moderno sin necesidad de instalación previa.

El sistema permite la gestión en tiempo real de las tareas de higiene, garantizando la trazabilidad completa de cada acción ejecutada por el personal de limpieza, al tiempo que proporciona a los supervisores herramientas avanzadas de control y auditoría.

1.2 Justificación y Problemática

En la actualidad, la gestión de la higiene en numerosos centros sanitarios depende de procesos manuales, hojas de registro en papel y comunicación verbal. Esta metodología introduce tres problemas críticos que SANICLEAR resuelve directamente:

🔍
Falta de Trazabilidad
Es imposible determinar con precisión quién limpió una zona específica y a qué hora exacta, dificultando auditorías sanitarias y acreditaciones.
⏱
Retraso en la Información
Las incidencias (habitación ocupada, falta de material biológico) no se comunican en tiempo real, generando cuellos de botella en la operativa.
📊
Dificultad de Auditoría
Analizar el rendimiento del personal o el consumo de productos requiere una costosa digitalización posterior de los registros físicos.

1.3 Ámbito del Proyecto

El sistema abarca cuatro módulos funcionales claramente diferenciados:

👥
Gestión de Usuarios y Roles
Control de acceso mediante RBAC (Role-Based Access Control) con dos perfiles diferenciados: Supervisor y Operario.
🏥
Gestión de Espacios (Zonas)
Catálogo digital de las áreas del hospital (quirófanos, habitaciones, pasillos), categorizadas por nivel de riesgo biológico y prioridad.
✅
Planificación de Tareas
Asignación dinámica de tareas de limpieza a los operarios con prioridad y fecha límite. Registro exacto del momento de finalización.
⚠️
Control de Incidencias
Sistema de reporte en tiempo real para que los operarios notifiquen problemas que impiden ejecutar una tarea.

Subsección 2.2

2.2 Backend y Base de Datos

SANICLEAR prescinde de un servidor backend tradicional (Node.js/Express, PHP, etc.) y adopta un modelo BaaS — Backend as a Service delegando toda la infraestructura en la plataforma Supabase. Esta decisión redujo el tiempo de desarrollo del backend estimado en un 40 % y permitió implementar seguridad a nivel de base de datos desde el primer día.

2.2.1 Tecnologías Utilizadas

Supabase (BaaS) PostgreSQL Supabase Auth (GoTrue) Row Level Security (RLS) PL/pgSQL — Triggers
ComponenteFunción en SANICLEAR
PostgreSQL Motor de base de datos relacional. Almacena todas las entidades del sistema (usuarios, zonas, tareas, incidencias) con integridad referencial garantizada.
Supabase Auth Servicio GoTrue que gestiona registro, inicio de sesión seguro y emisión de tokens JWT. El token contiene el rol del usuario y se verifica en cada petición.
Row Level Security (RLS) Políticas de seguridad a nivel de fila definidas en el motor de base de datos. Garantizan que un operario solo pueda leer sus propias tareas sin depender de validaciones en el código del cliente.
PL/pgSQL Triggers Lógica de negocio declarada directamente en la base de datos. El trigger on_auth_user_created asigna automáticamente el rol al nuevo usuario según su dominio de correo.

2.2.2 Entorno de Administración Backend

Supabase Dashboard SQL Editor integrado Table Editor visual

La gestión del esquema de base de datos se realiza íntegramente a través del Supabase Dashboard, una interfaz web que permite editar tablas, ejecutar sentencias SQL, gestionar usuarios registrados y visualizar o modificar las políticas de seguridad RLS de forma declarativa.

3.2.1 Automatización de Roles — Trigger

Para evitar la intervención manual de un administrador en cada alta de usuario, se implementó el siguiente trigger en PL/pgSQL que se ejecuta automáticamente al registrarse un nuevo usuario en auth.users:

PL/pgSQL — Trigger: on_auth_user_created
-- Función que se ejecuta tras cada INSERT en auth.users
CREATE OR REPLACE FUNCTION public.handle_new_user()
RETURNS trigger
LANGUAGE plpgsql
SECURITY DEFINER SET search_path = public
AS $$
DECLARE
  v_rol_id INT;
BEGIN
  -- Determina el rol según el dominio del correo electrónico
  IF NEW.email ILIKE '%admin%' OR NEW.email ILIKE '%supervisor%' THEN
    v_rol_id := 2; -- Rol Administrador/Supervisor
  ELSE
    v_rol_id := 1; -- Rol Operario (por defecto)
  END IF;

  -- Crea el perfil en la tabla pública de usuarios
  INSERT INTO public.usuarios (id_usuario, correo_electronico, id_rol, estado)
  VALUES (NEW.id, NEW.email, v_rol_id, 'activo');

  RETURN NEW;
END;
$$;

-- Vincula la función al evento de creación de usuario
CREATE TRIGGER on_auth_user_created
  AFTER INSERT ON auth.users
  FOR EACH ROW EXECUTE PROCEDURE public.handle_new_user();

3.2.2 Seguridad y Permisos — Row Level Security (RLS)

La seguridad de los datos no reside en el frontend (que podría ser manipulado), sino que se aplica directamente en el motor PostgreSQL mediante Row Level Security. Aunque el cliente realice una petición malintencionada, el servidor nunca devolverá datos que la política no autorice.

SQL — Políticas RLS aplicadas
-- Activar RLS en la tabla de tareas
ALTER TABLE public.tareas ENABLE ROW LEVEL SECURITY;

-- POLÍTICA 1: Un operario solo puede leer SUS tareas asignadas
CREATE POLICY "operario_lee_sus_tareas"
ON public.tareas FOR SELECT
USING (auth.uid() = id_operario);

-- POLÍTICA 2: Solo el administrador puede crear/modificar zonas
CREATE POLICY "admin_gestiona_zonas"
ON public.zonas FOR ALL
USING ((auth.jwt() ->> 'rol') = 'admin')
WITH CHECK ((auth.jwt() ->> 'rol') = 'admin');

-- POLÍTICA 3: Operario puede crear incidencias propias
CREATE POLICY "operario_crea_incidencias"
ON public.incidencias FOR INSERT
WITH CHECK (auth.uid() = id_operario);

Subsección 4.2

4.2 Configuración del Backend en Supabase

La estructura de la base de datos no se sube como archivos: se define ejecutando el script SQL de esquema directamente en el entorno de Supabase.

1
Crear el proyecto en Supabase
Acceder a supabase.com, crear una nueva organización y un proyecto. Elegir región europea (eu-central-1) para cumplir con el RGPD.
2
Ejecutar el script de esquema
Desde SQL Editor → New Query, pegar y ejecutar el archivo schema.sql del repositorio. El script crea las tablas, tipos ENUM, políticas RLS y triggers.
3
Obtener las credenciales de API
En Project Settings → API, copiar la URL del proyecto y la clave anónima anon key.
4
Configurar variables de entorno locales
Crear el archivo .env.local en la raíz del proyecto React:
.env.local — Variables de entorno (NO subir a Git)
# URL del proyecto Supabase — obtenida en Project Settings > API
VITE_SUPABASE_URL=https://xxxxxxxxxxx.supabase.co

# Clave pública anónima — NO es secreta, pero NO debe exponerse en repositorios públicos
VITE_SUPABASE_ANON_KEY=eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9.XXXXXXXX
⚠️
Seguridad: El archivo .env.local debe estar incluido en .gitignore para evitar exponer las claves en el repositorio público. Vercel inyectará estas variables a través de su panel de configuración en producción.
Subsección 4.3

4.3 Despliegue Automatizado del Frontend en Vercel

El flujo CI/CD automatizado elimina cualquier paso manual de publicación. El siguiente diagrama describe el proceso completo:

1
Push a la rama main
El desarrollador ejecuta git push origin main. Vercel detecta el evento via webhook de GitHub automáticamente.
2
Instalación de dependencias
Vercel ejecuta npm install en su entorno de build aislado, descargando todas las dependencias del package.json.
3
Compilación con Vite
Se ejecuta npm run build. Vite transpila el código JSX, elimina código no usado (tree-shaking) y genera archivos estáticos optimizados en /dist.
4
Publicación en la CDN Global
Los archivos de /dist se distribuyen en la CDN de Vercel (Edge Network), garantizando latencia mínima desde cualquier punto geográfico.
vercel.json — Configuración para SPA con React Router
{
  "rewrites": [
    {
      "source": "/(.*)",
      "destination": "/index.html"
    }
  ]
}
ℹ️
Nota sobre el rewrite: Este archivo es imprescindible para que las rutas de React Router (/dashboard, /tareas, etc.) funcionen correctamente al acceder directamente por URL. Sin él, Vercel devolvería un error 404 al intentar resolver rutas que no existen como ficheros físicos.
